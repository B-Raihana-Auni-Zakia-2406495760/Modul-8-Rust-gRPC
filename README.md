1. What are the key differences between unary, server streaming, and bi-directional streaming RPC methods, and in what scenarios would each be most suitable?
   Unary RPC merupakan model standar di mana satu request dari client akan dibalas dengan satu response dari server. Pendekatan ini paling cocok untuk operasi request response biasa seperti operasi CRUD pada database atau proses autentikasi. Server streaming adalah saat client mengirim satu request, tetapi server merespons dengan aliran data secara terus-menerus. Ini sangat cocok untuk mengunduh file besar atau menerima live feed data saham. Sedangkan bi-directional streaming memungkinkan client dan server saling mengirim aliran data secara bersamaan dan asinkron. Skenario yang paling pas untuk ini adalah aplikasi real time chat atau multiplayer game.

2. What are the potential security considerations involved in implementing a gRPC service in Rust, particularly regarding authentication, authorization, and data encryption?
   Untuk mengamankan komunikasi gRPC, data encryption wajib diimplementasikan menggunakan mTLS agar tidak terjadi man in the middle attack saat pertukaran data. Untuk authentication dan authorization, kita bisa memanfaatkan fitur interceptors di gRPC untuk mencegat setiap request dan memvalidasi token sebelum request tersebut diproses oleh logika service utama.

3. What are the potential challenges or issues that may arise when handling bidirectional streaming in Rust gRPC, especially in scenarios like chat applications?
   Tantangan utamanya adalah mengelola state dan concurrency. Di aplikasi chat, kita harus bisa mengelola daftar koneksi client yang sedang aktif secara aman di dalam sistem multithreading rust tanpa memicu race condition. Selain itu, kita harus menangani masalah ketika pesan masuk lebih cepat daripada kemampuan sistem memprosesnya dan menangani koneksi client yang terputus secara tiba-tiba tanpa membuat server crash atau memory leak.

4. What are the advantages and disadvantages of using tokio_stream::wrappers::ReceiverStream for streaming responses in Rust gRPC services?
   Keuntungan utamanya adalah ReceiverStream sangat mempermudah kita untuk mengubah mpsc::Receiver menjadi tipe Stream yang disyaratkan oleh tonic. Hal ini membuat integrasi arsitektur asinkron menjadi lancar. Kekurangannya adalah adanya overhead dari buffer channel. Jika ukuran buffer tidak disesuaikan dengan baik dan konsumen lambat membaca data, thread pengirim dapat terblokir atau data bisa terbuang.

5. In what ways could the Rust gRPC code be structured to facilitate code reuse and modularity, promoting maintainability and extensibility over time?
   Kode yang dihasilkan dari .proto sebaiknya ditaruh di module terpisah. Selain itu, business logic harus dipisahkan dari implementasi gRPC itu sendiri. Solusinya, kita bisa menggunakan traits di rust untuk melakukan abstraksi dan melakukan dependency injection. Hal ini akan sangat mempermudah unit testing dan maintainability kedepannya.

6. In the MyPaymentService implementation, what additional steps might be necessary to handle more complex payment processing logic?
   Untuk implementasi yang lebih solid, MyPaymentService harus diintegrasikan dengan modul database untuk menyimpan rekam jejak transaksi dan dihubungkan ke API payment gateway eksternal. Hal krusial lainnya adalah menambahkan mekanisme idempotency key untuk mencegah double charge ketika terjadi gangguan jaringan, serta menerapkan error handling yang mengembalikan status kode gRPC yang tepat untuk setiap kegagalan.

7. What impact does the adoption of gRPC as a communication protocol have on the overall architecture and design of distributed systems, particularly in terms of interoperability with other technologies and platforms?
   Penggunaan gRPC sangat memperkuat arsitektur microservices. Dengan protocol buffers sebagai contract, interoperability antar platform menjadi sangat mudah karena kita bisa generate kode client/server ke berbagai bahasa pemrograman tanpa khawatir ketidakcocokan tipe data. Dampaknya, komunikasi antar servis menjadi jauh lebih cepat, meskipun kita butuh infrastruktur load balancer yang harus mendukung HTTP/2 secara native.

8. What are the advantages and disadvantages of using HTTP/2, the underlying protocol for gRPC, compared to HTTP/1.1 or HTTP/1.1 with WebSocket for REST APIs?
   Keuntungan HTTP/2 adalah dukungan multiplexing, sehingga banyak request bisa lewat dalam satu koneksi TCP secara bersamaan tanpa head of line blocking. HTTP/2 juga menggunakan binary framing dan kompresi header yang menghemat bandwidth. Kekurangannya, karena formatnya binary, melakukan debugging menjadi lebih sulit jika dibandingkan dengan HTTP/1.1 yang berbasis plain text yang bisa langsung dibaca manusia.

9. How does the request-response model of REST APIs contrast with the bidirectional streaming capabilities of gRPC in terms of real-time communication and responsiveness?
   Model REST API pada dasarnya adalah stateless dan pull based. Untuk mendapat data baru, client harus terus-menerus melakukan polling yang tidak efisien dan sering ada latency. Sebaliknya, gRPC dengan bidirectional streaming menahan satu koneksi persisten terbuka. Hal ini memungkinkan server untuk melakukan push data langsung ke client tepat saat sebuah event terjadi. Otomatis, responsivitas aplikasi untuk real time communication menjadi jauh lebih tinggi dengan resource yang lebih hemat.

10. What are the implications of the schema-based approach of gRPC, using Protocol Buffers, compared to the more flexible, schema-less nature of JSON in REST API payloads?
    Implikasi utamanya adalah aplikasi menjadi lebih type safe dan ketat. Karena skema sudah didefinisikan secara eksplisit di file .proto, kemungkinan terjadinya runtime error akibat salah tipe data atau typo bisa diminimalisir. Serialisasi binary pada Protobuf juga jauh lebih cepat dan ringan dibanding melakukan parsing JSON. Kekurangannya hanyalah berkurangnya fleksibilitas. Setiap kali ada perubahan atribut, kita harus memperbarui file .proto dan recompile client dan server nya.
